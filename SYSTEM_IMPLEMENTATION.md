# WT99P4C5-S1 系统实现说明

本文面向不熟悉代码的读者，说明 WT99P4C5-S1 视频系统“用什么方法实现”。文中不展开具体函数和代码细节，而是从系统组成、数据流和控制逻辑解释整体工作方式。

## 1. 系统一句话说明

这套系统把 WT99P4C5-S1 开发板做成一个带网页管理能力的视频网关：

- 可以接 MIPI 摄像头或 USB 热像仪。
- 可以把实时画面变成 RTSP 视频流。
- 可以把照片和录像保存到 TF 卡。
- 可以通过网页查看状态、配置参数、拍照和管理媒体文件。
- 可以给外部 A3/Z-1mini 系统提供兼容接口和网络透传。
- 可以把 TCP 网络连接和串口互相转发。

## 2. 系统由哪些部分组成

从用户角度看，系统包含这些部分：

| 部分 | 作用 |
| --- | --- |
| WT99P4C5-S1 主控板 | 负责视频处理、网页、RTSP、TF 卡和串口透传 |
| ESP32-C5 Wi-Fi 协处理器 | 给主控板提供 Wi-Fi 能力 |
| MIPI 摄像头 | 提供普通可见光画面 |
| USB 热像仪 | 提供热成像原始数据 |
| TF 卡 | 保存照片和录像 |
| 网页 | 查看状态、配置参数和管理媒体 |
| RTSP 客户端 | 拉取实时视频流，例如播放器或上位机 |
| A3/Z-1mini 外部设备 | 通过兼容接口或网口透传接入系统 |

这些部分不是各自独立运行，而是围绕“视频帧”这条主线协同工作：

```text
摄像头/热像仪
  -> 生成图像帧
  -> 转成编码器能处理的格式
  -> 压缩成 H.264
  -> 送给 RTSP 客户端
  -> 需要录像时同时写入 MP4
  -> 需要拍照时抽取一帧保存 JPEG
```

## 3. 上电以后系统做什么

设备上电后，不会马上只启动摄像头，而是按“先把基础服务准备好，再按配置启动视频源”的顺序工作。

启动过程可以理解为：

1. 读取上次保存的配置。
2. 准备网络能力，让电脑、手机或外部设备能访问板子。
3. 尝试挂载 TF 卡。
4. 启动网页，让用户能查看和配置。
5. 启动 RTSP 服务，准备给客户端推送实时视频。
6. 启动 TCP-UART 服务，准备网络和串口互通。
7. 根据配置选择启动 MIPI 摄像头或 USB 热像仪。

这种启动方法的好处是：即使 TF 卡没有插好，网页、RTSP 和串口服务仍然可以启动，用户可以通过网页看到 TF 卡异常，而不是整机直接不可用。

## 4. 网络是怎么实现的

### 4.1 为什么需要 ESP32-C5

WT99P4C5-S1 使用 ESP32-P4 作为主要处理器。ESP32-P4 负责视频处理能力强，但不直接承担 Wi-Fi 射频工作。因此系统使用 ESP32-C5 作为 Wi-Fi 协处理器。

主控和 Wi-Fi 协处理器之间通过 SDIO 通信。可以把它理解成：

```text
ESP32-P4 主控
  <-> SDIO 高速通道
  <-> ESP32-C5 Wi-Fi 协处理器
  <-> Wi-Fi AP/STA
```

这种方法让主控专注于视频和存储，Wi-Fi 连接交给 C5 处理。

### 4.2 默认 Z-1mini 桥接模式

当前默认模式是 Z-1mini 网口透传。它的目标是让电脑连上 WT99 的 Wi-Fi 后，也能访问接在 WT99 以太网口上的 Z-1mini 吊舱。

工作方式如下：

```text
电脑或手机
  -> 连接 WT99 Wi-Fi AP
  -> WT99 收到 Wi-Fi 网络帧
  -> WT99 把需要转发的以太网帧送到板载以太网口
  -> Z-1mini 收到网络数据

Z-1mini 返回数据
  -> 进入 WT99 板载以太网口
  -> WT99 把以太网帧送回 Wi-Fi AP
  -> 电脑或手机收到数据
```

这不是普通路由转发，而是更接近“二层网桥”的方法。这样做可以让电脑、WT99 和 Z-1mini 看起来处在同一个 `192.168.144.0/24` 网段内。

默认地址：

- WT99 AP/本机服务：`192.168.144.2`
- Z-1mini 吊舱：`192.168.144.108`
- WT99 给 Wi-Fi 客户端分配地址范围：`192.168.144.20` 到 `192.168.144.60`

### 4.3 普通 AP/STA 模式

如果关闭 Z-1mini 桥接，系统也保留普通 Wi-Fi 模式：

- AP 模式：WT99 自己开热点，电脑或手机直接连 WT99。
- STA 模式：WT99 连接到已有路由器，成为局域网中的一个设备。

STA 模式有自动回退保护。切换到 STA 后，用户必须在新网络地址的网页上确认“能访问”。如果一段时间内没有确认，系统认为配置可能错误，会自动回到 AP 模式，避免设备失联。

## 5. 视频采集是怎么实现的

系统支持两种视频源，但一次只启用一种：

- MIPI 摄像头，用于普通可见光画面。
- USB 热像仪，用于热成像画面。

选择哪一种由网页配置决定，保存后重启生效。

### 5.1 MIPI 摄像头方法

MIPI 摄像头输出的是传感器图像数据。系统使用硬件视频通道接收这些数据，再经过图像处理和压缩。

流程如下：

```text
OV5647 摄像头
  -> MIPI-CSI 接收
  -> 图像处理单元整理颜色和格式
  -> 得到 YUV420 视频帧
  -> 硬件 H.264 编码
  -> RTSP 推流或录像
```

这种方法的重点是尽量使用硬件能力处理图像，减少 CPU 压力，保证实时性。

### 5.2 USB 热像仪方法

USB 热像仪输出的不是普通彩色视频，而是 Y16 原始热数据。每个像素是一个 16 位热值，数值越大代表热量越高。

系统不会直接把 Y16 发给播放器，而是在本机把热数据转换成可看的画面：

1. 接收 USB 热像仪的最新一帧 Y16 数据。
2. 统计这一帧中的最低热值和最高热值。
3. 把热值按比例拉伸到 `0 ~ 255` 的灰度范围。
4. 根据用户选择生成白热、黑热或铁红效果。
5. 转成 H.264 编码器需要的视频格式。
6. 推送 RTSP、保存录像或保存照片。

这种方法的优点是：

- 不依赖热像仪内部调色。
- 网页切换成像效果后，实时视频、录像和照片保持一致。
- 厂商私有 USB 控制命令可以关闭，减少阻塞和兼容风险。

## 6. RTSP 实时视频是怎么实现的

RTSP 可以理解成“播放器向设备要实时视频”的协议。系统采用 RTSP over TCP，也就是控制信令和视频数据都走 TCP 连接。

工作方式如下：

```text
播放器连接设备
  -> 发送 RTSP 请求
  -> 设备回复视频格式信息
  -> 播放器开始 PLAY
  -> 设备持续发送 H.264 视频帧
```

系统采用低延迟优先策略：

- 每个客户端只保留很短的视频队列。
- 如果网络发送变慢，系统丢弃旧帧，而不是让旧画面越积越多。
- 丢帧后等待新的关键帧，让播放器恢复完整画面。
- 第一个播放器开始播放时，系统启动视频采集。
- 最后一个播放器断开后，如果没有其他外部任务，系统暂停视频采集。

这种方法牺牲了“每一帧都必须送达”，换来实时画面不严重滞后。

## 7. 拍照是怎么实现的

拍照不是网页直接生成照片文件。网页只是发出“请拍一张”的请求，真正拍照发生在下一帧视频到来时。

流程如下：

```text
网页或外部接口请求拍照
  -> 系统检查 TF 卡和剩余空间
  -> 标记一次拍照请求
  -> 下一帧图像到达
  -> 拷贝这一帧图像
  -> 后台转换颜色格式
  -> 硬件 JPEG 编码
  -> 写入 TF 卡
```

这样设计的原因：

- 网页请求不会长时间卡住。
- 摄像头采集链路不会直接等待 TF 卡写文件。
- JPEG 编码和写卡放到后台处理，不拖慢实时推流。
- 如果 TF 卡不可用或空间不足，可以提前拒绝。

网页普通按钮当前会在有 RTSP 播放客户端时启用。底层接口也支持外部请求拍照时临时保持采集，等待下一帧图像到来。

## 8. 录像是怎么实现的

录像不是重新开一路视频编码，而是复用实时视频已经产生的 H.264 数据。

流程如下：

```text
外部控制请求开始录像
  -> 系统检查 TF 卡空间
  -> 视频源开始或保持采集
  -> 等待合适的 H.264 关键帧
  -> 创建 MP4 临时文件
  -> 持续写入 H.264 帧
  -> 达到分段时间后切新文件
  -> 停止录像时关闭文件并改成正式 MP4
```

这种方法的优点：

- 不需要第二路编码，节省 CPU、内存和编码器资源。
- RTSP 和录像画面来源一致。
- TF 卡写入变慢时，系统可以丢弃录像旁路中的旧帧，尽量不影响实时推流。
- 每个 MP4 分段从关键帧开始，播放器更容易正常打开。

当前网页普通界面没有录像启停按钮。录像主要由 A3 兼容接口控制。A3 控制开始录像时，系统会保持视频采集；A3 控制停止录像时，系统关闭当前录像。

## 9. TF 卡空间管理是怎么实现的

TF 卡空间管理分为两种情况。

第一种是新操作开始前：

- 拍照前检查空间，不足则拒绝拍照。
- 启动新录像前检查空间，不足则拒绝录像。

第二种是录像已经进行中：

- 如果切换新录像分段时空间不足，系统允许删除最早的旧 MP4。
- 自动删除只针对旧录像。
- 自动删除不会删除照片，也不会删除 TF 卡上其他无关文件。

这样做是为了避免录像长时间运行把卡写满，同时保护照片不被后台静默覆盖。

## 10. TF 卡文件怎么组织

系统按“上电会话”组织媒体文件。每次上电后，真正写入第一张照片或第一段录像时，系统创建一个会话目录。

目录结构如下：

```text
/sdcard/<会话目录>/photo/<时间>-<本次拍照序号>.jpeg
/sdcard/<会话目录>/video/<时间戳>_<分段号>.mp4
```

这种方法方便区分不同上电阶段产生的文件，也方便自动覆盖时按旧会话优先清理录像。

## 11. 网页是怎么实现的

网页不是放在外部服务器上，而是直接由 WT99 设备提供。电脑或手机连接到设备网络后，浏览器访问设备 IP 即可打开网页。

网页做三类事情：

1. 展示状态：当前时间、网络地址、RTSP 地址、TF 卡状态、容量、测速、客户端数。
2. 修改配置：视频源、分辨率、串口波特率、AP/STA 网络参数。
3. 管理媒体：查看照片、播放录像、拍照、删除文件。

网页和设备之间通过 HTTP 接口通信。网页定期读取设备状态，但不会每秒都请求设备。设备时间显示采用“设备给一个基准时间，浏览器本地自己走时，隔一段时间再校准”的方法，减少网络请求。

## 12. 配置保存是怎么实现的

用户在网页上修改的参数会保存到设备内部的非易失存储中。断电重启后，设备会读取这些参数继续工作。

系统对配置做了几类保护：

- 检查视频源是否合法。
- 检查 MIPI 分辨率档位是否被当前固件支持。
- 检查串口波特率范围。
- 检查 IP 地址文本格式。
- 检查 Wi-Fi 名称和密码长度。
- STA 模式需要网页确认，防止配置错误导致设备失联。

多数配置保存后需要重启生效。USB 热像仪成像效果是运行时参数，可以即时生效。

## 13. A3 兼容接口是怎么实现的

系统提供 A3 兼容接口，方便外部系统按固定格式控制 SD 卡状态、录像、拍照和时间。

主要方法：

- 外部系统查询 SD 卡状态，设备返回是否挂载、容量、剩余空间、是否可拍照、是否可录像。
- 外部系统下发录像参数，设备同步时间、检查 TF 卡、启动或停止录像。
- 外部系统要求拍照时，设备登记拍照请求，并在下一帧图像到来后保存照片。
- A3 兼容 RTSP 使用独立端口，方便外部系统按固定地址拉流。

当前兼容约束：

- 录像控制通道使用固定通道号 `1`。
- 兼容分段间隔为 `60` 秒。

## 14. TCP-UART 是怎么实现的

TCP-UART 的作用是把网络 TCP 连接和板子串口互相转发。

可以理解成：

```text
电脑 TCP 客户端
  <-> WT99 网络端口
  <-> WT99 串口
  <-> 外部串口设备
```

当前默认启动 UART0：

- UART0 TCP 端口：`8880`
- UART0 引脚：TX 37，RX 38

UART1 代码已经接入：

- UART1 TCP 端口：`8881`
- UART1 引脚：TX 4，RX 5

但 UART1 默认未在启动流程中启用。

## 15. 为什么要按需启动采集

视频采集、图像转换、H.264 编码都会消耗 CPU、内存、带宽和电源。系统采用按需采集：

- 有 RTSP 客户端播放时采集。
- 有 A3 录像任务时采集。
- 有外部拍照请求时短时间保持采集。
- 没有实时需求时暂停采集。

这样可以减少无用负载，同时仍然保证有人看视频、录像或拍照时有图像帧可用。

## 16. 系统的可靠性策略

系统在多个地方采用“实时链路优先”的策略：

- RTSP 发送堵塞时丢弃旧帧，避免画面越积越慢。
- 录像写卡慢时不反向阻塞实时视频。
- 拍照编码和写卡放到后台处理。
- TF 卡不可用时，本机网页和 RTSP 服务仍继续启动。
- STA 网络切换失败时自动回退 AP。
- 自动满卡覆盖只删除旧录像，不删除照片。

这些策略的目标是：在嵌入式资源有限、网络和 TF 卡状态不稳定的情况下，尽量保证设备可访问、实时画面可用、用户能看到明确状态。

## 17. 常见使用路径

### 17.1 查看 WT99 本机网页

电脑或手机连接 WT99 AP 后访问：

```text
http://192.168.144.2/
```

### 17.2 播放 WT99 本机视频

```text
rtsp://192.168.144.2
```

A3 兼容热成像地址：

```text
rtsp://192.168.144.2:580/live/6
```

### 17.3 访问 Z-1mini 吊舱视频

Z-1mini 通过 WT99 的桥接网络访问：

```text
rtsp://192.168.144.108
```

### 17.4 使用 TCP-UART0

```text
192.168.144.2:8880
```

## 18. 当前限制

- MIPI 摄像头和 USB 热像仪一次只启用一种。
- 网页普通界面当前不提供录像启停按钮。
- 录像主要由 A3 兼容控制接口启动和停止。
- USB 热像仪厂商私有调色命令默认关闭。
- UART1 透传默认未启动。
- Z-1mini 桥接模式启用时，普通以太网接入分支不会同时启动。
- AP 和 STA 不会同时启用。
- 多数配置需要重启后生效。
